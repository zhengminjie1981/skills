# 设计阶段检查清单

> 在开始实现之前，确保以下问题都已明确。

## 核心定位

- [ ] **单一职责**：这个 skill 只做一件事吗？
- [ ] **问题明确**：能清楚描述它解决什么问题吗？
- [ ] **用户价值**：用户为什么需要这个 skill？

## 触发场景

### 触发条件

- [ ] 列出至少 2 个具体触发场景
- [ ] 每个场景包含明确的关键词
- [ ] 关键词足够具体，不会与其他 skill 重叠

**示例**：
```
触发场景：
- "为...建立索引" → knowledge-index
- "转换文档为 Markdown" → doc2md
- "连接数据库" → db-mcp
```

### 不触发条件

- [ ] 列出至少 2 个不应触发的场景
- [ ] 说明为什么不触发

**示例**：
```
不触发场景：
- "查看单个文件" → 不是批量操作
- "复制文件" → 纯文件操作，不涉及索引
```

## 资源规划

### 脚本 (scripts/)

- [ ] 确定是否需要脚本
- [ ] 列出需要的脚本及其功能

**需要脚本的场景**：
- 需要确定性执行
- 需要调用外部工具
- 逻辑复杂，AI 难以一次写对

### 文档 (references/)

- [ ] 规划 references/ 目录结构
- [ ] 确定哪些内容放入 SKILL.md，哪些放入 references/

**划分原则**：
- SKILL.md：快速开始、决策树、核心功能表
- references/：详细规范、配置说明、故障排除

### 资源 (assets/)

- [ ] 确定是否需要 assets/
- [ ] 列出需要的资源文件

**需要 assets 的场景**：
- 需要模板文件
- 需要图片/图标
- 需要配置模板

## 渐进式加载设计

- [ ] SKILL.md 核心内容 < 500 行
- [ ] references/ 按主题组织
- [ ] 在 SKILL.md 中提供清晰的加载指引

**加载层级**：
1. Front matter（metadata）→ 总是加载
2. SKILL.md body → skill 触发时加载
3. references/ → 按需加载
