# Task Stack - 设计文档

## 问题

复杂开发任务中存在三种上下文丢失场景：

1. **链式分支**：修 A → 发现要先修 B → 修 B 发现要修 C。分支链展开后忘了 A 是什么。
2. **会话中断**：Claude Code 会话结束或意外中断后，所有工作上下文丢失。
3. **话题漂移**：工作中被其他话题带偏，原任务被遗忘。

现有机制的缺口：

| 机制 | 覆盖范围 | 缺口 |
|------|---------|------|
| TaskCreate/TaskList | 会话内任务追踪 | 不跨会话持久化 |
| Memory | 跨会话持久化 | 无栈语义，不适合任务管理 |
| Plan 模式 | 任务拆解 | 一次性，不跟踪执行 |

v1 额外暴露出两个使用问题：

1. `.tasks/stack.json` 同时保存索引、状态和完整任务详情，更新 `resume_hint` 也需要改写大 JSON。
2. “记录后续任务”和“临时插入任务”语义不够清晰，容易把不应打断当前任务的待办记录成任务切换。

v2 拆分为每个任务一个 JSON 后解决了大文件重写问题，但对尚未开始的 backlog 任务过重。很多待办只需要标题、优先级和来源，不需要完整详情文件。

## 方案

### 核心思路

将任务管理建模为**栈**而非列表：

- **Push**：新任务入栈，当前任务自动挂起
- **Pop**：任务完成出栈，恢复上级任务
- **栈顶**：始终是当前活跃任务

栈语义天然匹配“先处理分支，再回主线”的工作模式。

v2.1 在保留栈语义的基础上采用混合模型：

- `stack.json` 是执行流索引，用于快速找到 active 和栈顺序
- `backlog.jsonl` 是轻量待办列表，一行一个尚未开始的任务
- `tasks/*.json` 只保存 active / suspended / blocked 等已经进入执行流、需要恢复上下文的任务

### 数据持久化

任务数据存储在项目根目录 `.tasks/` 下，gitignored：

```text
.tasks/
├── stack.json              # 执行流索引：active、stack、next_id
├── backlog.jsonl           # 轻量待办：一行一个尚未开始的任务
├── tasks/
│   ├── T001.json           # active / suspended / blocked 任务详情
│   └── T003.json
└── archive.json            # 已完成任务归档摘要
```

选择项目根而非用户目录，因为任务上下文是项目级的。

### 为什么使用混合模型

v1 的单文件结构简单，但实际使用中会产生大文件频繁重写：

- 只更新一个任务的 `resume_hint`，也要改写整个 `stack.json`
- 追加 backlog 任务会触碰完整任务栈
- 挂起任务时需要同时改写当前任务和新任务的大对象

纯 v2 的“每个任务一个 JSON”又会让 backlog 变重：

- 尚未开始的简单待办也会创建独立文件
- `.tasks/tasks/` 容易堆积大量低价值文件
- list backlog 需要读取多个文件，或在索引中重复摘要字段

v2.1 的混合模型按任务生命周期分层：

- **尚未开始**：只写 `backlog.jsonl`，一行足够识别和排序
- **进入执行流**：创建 `tasks/<id>.json`，保存详情、状态和恢复上下文
- **被挂起**：必须写入 `progress_snapshot`，确保跨会话恢复
- **完成**：写入 `archive.json` 摘要

这样既减少大 JSON 重写，又避免为简单待办创建过多文件。

### 两类任务切换语义

#### 记录后续任务：不打断

用户说“后续还要做”“先记一下”“待办”时，应执行 `add` / `later`：

- 向 `backlog.jsonl` 追加一个或多个轻量待办
- 默认不创建任务详情 JSON
- 不改变 `active`
- 不修改当前 active 任务状态
- 不打断当前上下文

如果用户给出的待办本身包含大量上下文，可以创建 `tasks/<id>.json` 保存长信息，并在 backlog 行中用 `details` 指向它。

#### 临时插入任务：先快照再打断

用户说“先处理这个”“把当前任务延后”“临时插入任务”时，应执行 `push` / `interrupt`：

1. 使用 Agent 当前上下文总结被打断任务的进展和关键信息。
2. 如当前任务尚无详情文件但已有较多上下文，先创建 `tasks/<id>.json`。
3. 写入当前任务的 `resume_hint` 和 `progress_snapshot`。
4. 将当前任务标记为 `suspended`。
5. 创建或激活新任务并放到栈顶。

这个顺序是 v2.1 的关键不变量：**先快照，再挂起，再开始新任务**。

### progress_snapshot

`progress_snapshot` 是为了避免用户在任务切换时重复记录上下文。

它保存：

- 当前任务做到哪一步
- 已完成的关键操作
- 已做出的判断或决策
- 相关文件、命令、错误、测试结果
- 下一步原本准备做什么
- 为什么被打断

如果 Agent 当前上下文不足，只问最小必要澄清问题，不要求用户完整复述进展。

### stack.json 与 backlog.jsonl 的信息充分性

`stack.json` 和 `backlog.jsonl` 不承担完整任务信息保存职责，只承担索引、展示和路由职责。

`stack.json` 应回答：

- 当前任务是谁
- 执行流栈顺序是什么
- 下一个 id 是多少
- 任务详情去哪读

`backlog.jsonl` 应回答：

- 有哪些后续任务
- 大概是什么
- 优先级如何
- 什么时候记录的
- 必要时去哪找更多信息

因此 backlog 行保留 `id`、`title`、`priority`、`created` 和可选 `source`，长上下文进入任务详情 JSON。

### 会话启动提醒

Skill 本身是按需加载的，无法在会话启动时自动触发。通过 CLAUDE.md 中的一条规则解决：

> 如果 `.tasks/stack.json` 存在，简要告知用户当前活跃/挂起任务

v2.1 下这层轻量检查只需：

1. 读取 `.tasks/stack.json`
2. 加载 active 任务文件
3. 按需读取 suspended 任务详情和 `backlog.jsonl` 摘要

如果检测到 v1 完整对象格式，只提示可迁移，不在启动检查中自动迁移。

### 不使用脚本的原因

任务栈操作仍是简单的 JSON / JSONL 文档读写，Claude 可以直接用 Read/Write/Edit 工具完成。不需要额外脚本层。

v1 到 v2.1 的转换也先采用提示词驱动的 `migrate` 操作：

- 更容易在规范未完全稳定时调整
- 遇到冲突可以交互式确认
- 避免维护迁移脚本和校验脚本

后续只有在真实使用中出现频繁迁移、格式损坏或批量校验需求时，才考虑新增脚本。

### 为什么做成 Skill

Skill 机制适合这个需求的原因：

1. **行为规范可共享**：Skill 文件提交到 git，团队成员都能使用
2. **运行状态私有**：`.tasks/` 被 gitignore，每个人的任务栈独立
3. **结构化**：SKILL.md 集中定义所有操作规范，比散落在 Memory 中的规则更可靠
4. **可演进**：和项目代码一起维护，可以版本化迭代

## 与其他方案的对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **纯 Memory** | 无额外文件 | 无结构化格式，行为规范散落多处 |
| **CLAUDE.md 规则** | 始终加载 | CLAUDE.md 不适合放详细规范，会膨胀 |
| **v1 单 stack.json** | 简单直观 | 高频更新会造成大 JSON 重写 |
| **v2 每任务 JSON** | 更新粒度小，恢复上下文清晰 | backlog 过重，简单待办也产生文件 |
| **v2.1 混合模型** | 待办轻量，执行流可恢复，更新粒度小 | 需要同时维护 JSON 和 JSONL |
| **Skill + CLAUDE.md** | 结构化 + 启动提醒 | 两处维护（但 CLAUDE.md 只有一行） |

选择 Skill + CLAUDE.md + v2.1 混合模型：Skill 定义详细行为，CLAUDE.md 提供启动入口，`.tasks/` 保存用户私有运行状态。

## 未来扩展

可能的后续迭代方向：

- 自动迁移脚本
- JSON / JSONL schema 校验脚本
- 任务时间统计
- 定时提醒（结合 CronCreate）
- 团队共享任务（取消 gitignore，按人分文件）
- 任务依赖图或 blocked-by 关系
