# 开发工作流检查清单

## 新模块开发检查清单

### Phase 1: 需求分析

- [ ] **功能边界明确**
  - 定义模块职责（做什么/不做什么）
  - 确认上游输入数据结构
  - 确认下游输出数据结构

- [ ] **依赖关系梳理**
  - 列出依赖的上游模块
  - 列出依赖的下游模块
  - 确认依赖的数据结构/接口是否已定义

### Phase 2: 架构设计（如需）

- [ ] **系统架构更新**
  - 更新系统架构设计文档
  - 明确模块在系统中的位置
  - 定义模块边界和数据流

- [ ] **架构评审**
  - 提交架构设计评审
  - 记录评审意见
  - 根据评审意见修改

### Phase 3: 规范定义（如需）

- [ ] **数据结构规范**
  - 定义新增数据结构
  - 更新已有数据结构（如有修改）
  - 检查与现有数据结构的一致性

- [ ] **接口规范**
  - 定义新增接口
  - 更新已有接口（如有修改）
  - 检查与现有接口的一致性

### Phase 4: 模块设计

- [ ] **模块设计文档**
  - 模块定位与边界
  - 输入输出数据规范（引用数据结构规范）
  - 核心算法/逻辑设计
  - 接口设计（引用接口规范）
  - 验证方案与测试用例
  - 已知约束与改进路线图
  - 版本历史

- [ ] **设计评审**
  - 提交模块设计评审
  - 记录评审意见
  - 根据评审意见修改

### Phase 5: 文档一致性检查

- [ ] **架构一致性**
  - 模块边界与系统架构设计一致
  - 数据流与系统架构设计一致

- [ ] **数据结构一致性**
  - 模块文档引用的数据结构与规范一致
  - 无重复定义（只引用，不复制）

- [ ] **接口一致性**
  - 模块文档中的接口定义与规范一致
  - 接口签名一致（参数、返回值）

- [ ] **一致性评分**
  - 评分 ≥ 95%
  - 记录不一致项并修复

### Phase 6: 代码实现

- [ ] **编码规范**
  - 代码风格符合项目规范
  - 代码注释引用对应文档章节
  - 无硬编码的魔法数字/字符串

- [ ] **单元测试**
  - 测试覆盖率 ≥ 80%
  - 所有测试用例通过
  - 测试用例与文档定义一致

- [ ] **代码审查**
  - 提交代码审查
  - 处理审查意见
  - 审查通过

### Phase 7: 验证与收尾

- [ ] **集成测试**（如涉及接口）
  - 与上游模块集成测试
  - 与下游模块集成测试
  - 集成测试通过

- [ ] **文档更新**
  - 补充实测结果到模块设计文档
  - 更新 README 中的模块状态
  - 更新项目计划与进展记录

- [ ] **清理工作**
  - 清理临时文件
  - 更新文档索引
  - 提交最终版本

## 迭代变更检查清单

### Phase 1: 变更发起

- [ ] **变更内容明确**
  - 变更原因
  - 变更范围（哪个模块、哪个数据结构/接口）
  - 变更类型（数据结构变更/接口变更/算法变更/参数变更）

- [ ] **变更记录**
  - 在模块设计文档中记录变更需求
  - 在项目计划与进展记录中记录变更计划

### Phase 2: 影响分析

- [ ] **文档影响分析**
  - 列出所有受影响的文档
    - [ ] 系统架构设计（如模块边界/数据流变更）
    - [ ] 数据结构规范（如数据结构变更）
    - [ ] 接口规范（如接口变更）
    - [ ] 对应模块设计文档
    - [ ] 关联的其他模块设计文档
    - [ ] 验证方案文档（如测试用例受影响）

- [ ] **代码影响分析**
  - 列出所有受影响的代码文件
    - [ ] 源代码文件
    - [ ] 测试文件
    - [ ] 配置文件
    - [ ] 脚本文件

### Phase 3: 更新源头文档

- [ ] **更新优先级**
  - 1. 系统架构设计（如涉及）
  - 2. 数据结构规范或接口规范（如涉及）
  - 3. 对应模块设计文档

- [ ] **版本历史更新**
  - 在受影响文档的版本历史中追加变更记录
  - 变更记录格式：`| vX.Y | YYYY-MM-DD | [变更类型] 变更描述。影响范围：... |`

### Phase 4: 更新关联文档

- [ ] **逐个更新**
  - 按影响分析结果，逐一更新受影响的文档
  - 重点检查：数据结构字段名/类型、接口签名

- [ ] **引用一致性**
  - 确保所有文档引用的是更新后的规范
  - 无过时的引用

### Phase 5: 文档一致性检查

- [ ] **架构一致性**
  - 变更后的模块边界与系统架构设计一致
  - 变更后的数据流与系统架构设计一致

- [ ] **数据结构一致性**
  - 所有模块引用的数据结构与更新后的规范一致

- [ ] **接口一致性**
  - 所有模块的接口定义与更新后的规范一致

- [ ] **一致性评分**
  - 评分 ≥ 95%
  - 记录并修复不一致项

### Phase 6: 代码修改

- [ ] **按文档修改代码**
  - 严格按更新后的文档修改代码
  - 代码注释同步更新

- [ ] **测试文件更新**
  - 同步更新单元测试
  - 同步更新集成测试（如涉及）

- [ ] **测试验证**
  - 单元测试通过
  - 集成测试通过（如涉及）
  - 回归测试通过（重要变更）

### Phase 7: 验证与收尾

- [ ] **验证变更**
  - 运行完整测试套件
  - 确认变更无副作用
  - 性能测试（如涉及性能变更）

- [ ] **文档更新**
  - 更新项目计划与进展记录
  - 更新 README 中的相关状态或指标
  - 更新文档索引（如有）

- [ ] **变更通知**
  - 通知相关团队成员
  - 记录变更日志

## 定期维护检查清单

### 每日检查（开发期间）

- [ ] 文档与代码一致

- [ ] Git 提交信息清晰
- [ ] 工作进展已记录

- [ ] 临时目录清理（删除超过7天的文件）
- [ ] 文档一致性检查
- [ ] 测试覆盖率检查（≥ 80%）
- [ ] 项目计划与进展记录更新

### 每月检查

- [ ] 完整文档一致性审查
- [ ] 版本历史完整性检查
- [ ] 文档索引更新
- [ ] README 状态更新
- [ ] 清理过期的临时文件和分支
