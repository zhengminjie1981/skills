# 开发工作流规范

> **适用范围**：所有软件开发项目
> **强制约束**：所有代码实现必须有对应文档先行，不允许无文档直接编码

---

## 1. 总体原则

| 编号 | 原则 | 核心要求 |
|------|------|----------|
| P-01 | **文档优先** | 任何模块或功能的实现，必须先完成技术文档，再编写代码 |
| P-02 | **变更留档** | 任何迭代修改，必须先更新相关文档并完成影响分析，再修改代码 |
| P-03 | **模块化边界** | 模块间通过数据结构规范和接口规范进行交互，禁止跨模块直接访问内部实现 |
| P-04 | **一致性维护** | 文档更新后，须同步检查并更新所有引用该内容的文档 |
| P-05 | **集中进展记录** | 所有工作计划与进展统一记录在项目计划与进展记录中；各模块设计文档只保留结构性的版本历史，不重复记录进展 |
| P-06 | **文档定位清晰** | 每份文档须明确自身层级、回答的问题及上下游关联，详见 §1.1 |
| P-07 | **文档集中管理** | 所有正式文档须集中存放在统一的文档目录下，文档目录名称可由项目决定（如 `doc/`、`docs/` 等），便于管理和维护 |
| P-08 | **临时文件隔离** | 临时文档、脚本和数据须放在专门的临时目录下（如 `temp/`、`tmp/` 等），定期清理，避免污染正式文档 |
| P-09 | **版本可追溯** | 所有文档必须包含版本历史章节，记录变更内容、日期和影响范围；使用语义化版本号 |

### 1.1 文档层级体系与定位规范

本项目文档按以下层级组织，**每份文档必须且只能属于一个层级**：

| 层级 | 目录 | 回答的问题 | 典型特征 |
|------|------|-----------|---------|
| **L1 系统级** | `系统设计/` 或 `architecture/` | 系统做什么、整体技术路线、模块边界怎么划分 | 读者是架构师；不包含实现代码；结论驱动下级文档 |
| **L2 规范级** | `技术规范/` 或 `specs/` | 跨模块数据结构与接口的权威定义 | 单一事实来源；所有模块引用而非复制 |
| **L3 模块级** | `模块设计/` 或 `design/` | 某个模块如何实现 | 聚焦单一模块职责边界；引用 L2 规范 |
| **L4 验证级** | `测试验证/` 或 `validation/` | 技术方案的验证方法、测试策略与性能评估 | 回答"如何验证"；包含测试计划、性能测试、安全测试等；结论上移至 L1/L2 |
| **L5 报告级** | `报告/` 或 `reports/` | 某次实验或阶段工作的结果记录 | 事后产物；只读；链接到决策文档 |
| **临时** | `temp/` 或 `tmp/` | 临时文档、脚本、数据 | 临时产物；定期清理；不纳入版本管理 |

**新建文档时的强制流程**：

```
1. 确定层级 → 放入对应目录
2. 在文档头部写明"文档定位"：
   > **文档定位**：本文档属于 [层级]，回答"[核心问题]"。
   > - 上位文档：[链接]
   > - 下游文档：[链接]（若有）
3. 不得将低层级内容（如实现代码、参数调优细节）嵌入高层级文档
4. 已被吸收的内容（结论已上移至 L1/L2）应从低层级文档中删除或标注"见[目标文档]"
```

**层级错位的典型反模式**（禁止）：

| 反模式 | 示例 | 正确做法 |
|--------|------|---------|
| L1 文档包含实现代码 | 系统总体方案.md 含具体算法实现代码 | 代码移至 L4，总体方案只写结论 |
| L3 文档重复 L2 数据结构定义 | 模块设计.md 内嵌数据结构定义 | 引用数据结构规范文档 |
| L4 文档堆叠在 L1 目录 | 验证实施计划放在系统设计/ | 移入算法研究/ 或 reports/ |
| 多个文档复制同一代码块 | 同一段代码出现3次 | 只在权威来源处保留，其他处引用 |
| 临时文件混入正式目录 | 调试脚本放在 src/ 目录 | 移入 temp/ 或 scripts/temp/ |

**文档组织原则**：

- 所有正式文档必须集中存放在统一的文档目录下（目录名称由项目决定）
- 临时文件（实验脚本、调试数据、草稿文档）必须放入临时目录，定期清理
- 项目计划与进展记录是唯一的进展记录位置，各模块文档不得重复记录
- 文档目录下必须包含导航入口文件（如 `README.md` 或 `INDEX.md`）

**强制要求的目录**：

| 目录类型 | 强制要求 | 说明 |
|---------|---------|------|
| 文档根目录 | ✅ 必须存在 | 名称可自定义（如 `doc/`、`docs/`），所有正式文档必须在此目录下 |
| 临时目录 | ✅ 必须存在 | 名称可自定义（如 `temp/`、`tmp/`），并在 `.gitignore` 中排除 |
| 文档导航入口 | ✅ 必须存在 | 位于文档根目录下，如 `doc/README.md`，提供文档索引和导航 |
| 项目进展记录 | ✅ 必须存在 | 位于文档根目录下（或子目录），记录唯一的进展跟踪 |

**文档目录结构示例**（可根据项目灵活调整）：

```
project/
├── doc/                          ← 文档根目录（名称可自定义）
│   ├── README.md                 ← 文档导航入口（必须）
│   ├── 项目计划与进展记录.md      ← 进展记录（必须）
│   │
│   ├── 01-系统设计/              ← L1 系统级文档
│   │   ├── 系统架构设计.md
│   │   └── 技术选型.md
│   │
│   ├── 02-技术规范/              ← L2 规范级文档
│   │   ├── 数据结构规范.md
│   │   ├── 接口规范.md
│   │   └── 编码规范.md
│   │
│   ├── 03-模块设计/              ← L3 模块级文档
│   │   ├── 用户认证模块.md
│   │   └── 数据处理模块.md
│   │
│   ├── 04-测试验证/              ← L4 验证级文档
│   │   ├── 测试计划.md
│   │   ├── 性能测试报告.md
│   │   └── 安全测试报告.md
│   │
│   ├── 05-报告/                  ← L5 报告级文档
│   │   ├── 2026-03-里程碑1总结.md
│   │   └── 2026-04-技术调研报告.md
│   │
│   ├── templates/                ← 文档模板（可选）
│   │   ├── 模块设计模板.md
│   │   └── 测试计划模板.md
│   │
│   └── archive/                  ← 归档文档（可选）
│       └── 2025-Q4/
│           └── 已废弃功能设计.md
│
├── temp/                         ← 临时目录（必须，名称可自定义）
│   ├── scripts/                  ← 临时脚本
│   ├── data/                     ← 临时数据
│   └── drafts/                   ← 草稿文档
│
└── src/                          ← 源代码（不含文档）
```

**说明**：
- L1-L5 的子目录名称和层级可根据项目实际情况灵活调整
- 可根据项目发展阶段动态调整目录结构（如初期可简化为"设计文档/"和"报告/"）
- `templates/` 和 `archive/` 目录为可选，根据项目需要决定是否创建
- 目录可以使用中文或英文，但项目内必须统一，不得混用

## 2. 项目模块划分与数据接口

### 2.1 模块层次结构示例

```
src/
├── features/          【特征提取模块】
│   └── 接收：FeatureExtractionInput
│       输出：FeatureExtractionResult
│
├── matching/          【特征匹配模块】
│   ├── feature_matcher    接收：两帧特征 → 输出：MatchResult
│   ├── feature_tracker    接收：当前帧 + 历史帧数据 → 输出：追踪数据
│   └── integrated_matcher 接收：特征 → 输出：综合匹配结果
│
├── localization/      【定位计算模块】
│   └── 接收：匹配结果 + 地图数据 → 输出：位姿估计
│
├── visualization/     【可视化模块】
│   └── 接收：任意模块输出（只读，不修改数据）
│
└── utils/             【工具模块】
    └── 无上游依赖，被其他模块引用
```

### 2.2 跨模块数据结构规范

所有模块间数据交换**必须且只能**使用数据结构规范中已定义的数据结构。不允许以字典、裸元组等临时格式作为模块间的接口类型。

**核心要求**：
- 数据结构定义统一放在数据结构规范文档中
- 接口定义统一放在接口规范文档中
- 模块设计文档引用规范文档，而非复制定义

## 4. 迭代变更工作流

**黄金规则**：没有变更文档，不改代码。

### 4.1 触发条件

以下情况必须按本节流程处理：
- 修改已有数据结构的字段（增删改）
- 修改已有接口的签名（参数、返回值）
- 修改核心算法逻辑或参数
- 添加/删除模块文件
- 修改模块间的数据流向

### 4.2 变更流程

```
┌─────────────────────────────────────────────────────┐
│ Step 1：变更发起                                    │
│   - 明确变更内容和变更原因                          │
│   - 确认变更范围（哪个模块、哪个数据结构/接口）     │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│ Step 2：影响分析                                    │
│   参考文档一致性检查中的依赖关系图                  │
│   列出所有受影响的文档和代码文件：                  │
│   ① 系统架构设计（若模块边界/数据流变更）           │
│   ② 数据结构规范（若数据结构变更）                  │
│   ③ 接口规范（若接口变更）                         │
│   ④ 使用该数据结构/接口的所有模块设计文档           │
│   ⑤ 相关测试文件（tests/）                         │
│   ⑥ 相关脚本（scripts/）                           │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│ Step 3：更新源头文档                                │
│   - 优先更新系统架构设计（如涉及模块边界/数据流）   │
│   - 然后更新数据结构规范或接口规范                  │
│   - 在对应模块设计文档的版本历史章节追加变更条目    │
│   - 在项目计划与进展记录中追加本次变更的工作计划    │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│ Step 4：更新关联文档                                │
│   - 按影响分析结果，逐一更新受影响的文档            │
│   - 重点检查：数据结构字段名/类型、接口签名一致性   │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│ Step 5：文档一致性检查（编码前检查点）              │
│   - 执行文档一致性检查中的检查清单                  │
│   - 确认一致性评分 ≥ 95% 后方可进入编码             │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│ Step 6：代码修改                                    │
│   - 按更新后的文档修改代码                          │
│   - 同步更新测试文件（tests/）                      │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│ Step 7：验证与收尾                                  │
│   - 运行测试，确认变更无副作用                      │
│   - 在项目计划与进展记录中追加本次变更的完成情况    │
│   - 更新README中的相关状态或指标                    │
│   - 更新文档索引（如有）                            │
└─────────────────────────────────────────────────────┘
```

### 4.3 变更记录格式

在受影响的模块设计文档的"版本历史"章节追加一行：

```
| vX.Y | YYYY-MM-DD | [变更类型] 变更描述。影响范围：列出受影响模块 |
```

变更类型标签：`[数据结构变更]` `[接口变更]` `[算法变更]` `[参数变更]` `[文档修正]` `[架构变更]`

## 5. 文档一致性维护规范

### 5.1 文档层次与权威性

```
系统架构设计     ◄── 最高权威（模块边界、数据流、架构决策的唯一真实来源）
      │
      │ 驱动
      ▼
数据结构规范     ◄── 次高权威（数据结构的唯一真实来源）
接口规范         ◄── 次高权威（接口的唯一真实来源）
      │
      │ 引用
      ▼
各模块设计文档   ◄── 模块级权威
      │
      │ 引用
      ▼
验证方案/报告    ◄── 过程记录
      │
      │ 汇总
      ▼
README / 文档索引  ◄── 面向导航的索引文档
```

**规则**：下层文档与上层文档冲突时，**上层文档为准**，须更新下层文档以消除冲突。

**开发顺序**：系统架构设计 → 数据结构规范/接口规范 → 模块设计文档 → 代码实现

### 5.2 定期一致性检查

每完成一个开发阶段（或每3天），执行以下检查：

| 检查项 | 检查内容 | 参考文档 |
|--------|----------|----------|
| 架构一致性 | 所有模块边界和数据流与系统架构设计一致 | 系统架构设计 |
| 数据结构一致性 | 所有模块文档中引用的数据结构字段与数据结构规范一致 | 数据结构规范 |
| 接口签名一致性 | 所有模块文档中的接口定义与接口规范一致 | 接口规范 |
| 状态一致性 | README中的模块状态表与实际代码状态一致 | README |
| 进展一致性 | 项目计划与进展记录中的完成情况与成果物清单一致 | 项目计划与进展记录 |
| 索引完整性 | 文档索引中列出了所有现有文档 | 文档索引 |
| 临时文件隔离 | 临时文件未混入正式文档或代码目录 | 项目目录结构 |

检查结果记录在文档一致性检查记录中。

### 5.3 文档更新顺序

当需要更新多个文档时，按以下顺序进行（从源头到末端）：

```
1. 系统架构设计（如模块边界或数据流变更）
2. 数据结构规范或接口规范（如涉及）
3. 对应模块设计文档
4. 关联的其他模块设计文档
5. 验证方案文档（如测试用例受影响）
6. 项目计划与进展记录（记录本次工作计划与进展，唯一进展记录位置）
7. README / 文档索引（汇总层）
```

## 5.4 临时文件管理规范

**临时文件定义**：
- 调试脚本、测试数据、性能分析数据
- 草稿文档、实验性代码片段
- 中间产物、临时导出文件

**强制要求**：
- 临时文件必须放在临时目录（如 `temp/`），并在 `.gitignore` 中排除
- 定期清理（建议每周），超过 7 天的临时文件应删除或归档

**临时文件迁移流程**：

```
临时文件 → 评估 → 归档或删除

1. 草稿文档完成
   ↓
   移动到对应正式目录
   例如: temp/drafts/用户认证.md → doc/03-模块设计/用户认证模块.md
   并补充完整的文档结构（版本历史等）

2. 临时验证报告
   ↓
   移动到报告目录并重命名（加日期前缀）
   例如: temp/validation_report.md → doc/05-报告/2026-03-02-验证报告.md

3. 临时脚本
   ↓
   评估是否需要保留
   - 需要保留 → 移动到 scripts/ 目录
   - 不需要 → 删除
```

**临时文件清理脚本**：

```bash
# 清理超过7天的临时文件
find temp/ -mtime +7 -type f -delete

# 列出将要删除的文件（先预览）
find temp/ -mtime +7 -type f

# 清理空目录
find temp/ -type d -empty -delete
```

## 6. 禁止行为清单

以下行为违反本规范，在任何情况下均不允许：

- [ ] 直接编写代码而未创建或更新对应技术文档
- [ ] 修改数据结构字段而未同步更新数据结构规范
- [ ] 修改接口签名而未同步更新接口规范
- [ ] 在模块间使用裸字典/元组代替规范数据结构传递数据
- [ ] 创建新数据结构而不在数据结构规范中注册
- [ ] 跳过影响分析直接修改代码
- [ ] 文档与代码不一致超过24小时
- [ ] 在模块设计文档或其他文档中重复记录工作进展（进展统一记录在项目计划与进展记录中）
- [ ] 将文档分散在项目各处而不集中在文档目录
- [ ] 将临时文件混入正式文档或代码目录
- [ ] 不经过系统架构设计直接定义数据结构或接口
- [ ] 在低层级文档中复制而非引用高层级文档内容
- [ ] 文档缺少版本历史或版本历史不完整
- [ ] 使用临时文件（temp/）进行长期存储而不迁移到正式目录
- [ ] 删除文档而不归档或保留版本历史
- [ ] 缺少文档导航入口（如 doc/README.md）

def 开发新模块(模块名, 功能描述):
    """AI 执行新模块开发的决策流程"""

    # Step 1: 检查是否需要创建文档
    if not exists(f"doc/03-模块设计/{模块名}.md"):

        # Step 2: 分析输入输出数据结构

        for ds in 需要的数据结构:

            print("  3. 最后再开始编码")

        # Step 3: 使用模板创建文档
        print("📝 正在创建模块设计文档...")
        模板 = read("doc/templates/模块设计模板.md") or read("dev-workflow/references/templates.md")

            "日期": get_current_date()
        }
        创建文档(f"doc/03-模块设计/{模块名}.md", 模板, 填充内容)

        print(f"✅ 已创建 doc/03-模块设计/{模块名}.md")
        print("文档包含以下章节：")
        print("  1. 模块定位与边界")
        print("  2. 输入输出数据规范")
        print("  3. 核心算法/逻辑设计（待补充）")
        print("  4. 接口设计（待补充）")
        print("  5. 验证方案与测试用例（待补充）")
        print("\n请先完善文档，特别是'核心算法'章节，然后我再开始编码。")
        return "等待用户完善文档"

    # Step 4: 文档已存在，检查完整性
    文档内容 = read(f"doc/03-模块设计/{模块名}.md")
    必需章节 = ["模块定位与边界", "输入输出数据规范", "核心算法", "验证方案", "版本历史"]

    for 章节 in 必需章节:

        print("请先补充这些章节后再开始编码")

    # Step 5: 文档完整，开始编码

    return "开始编码"

### 8.2 迭代变更流程
```python
def 修改代码(文件路径, 变更内容):
    """AI 执行代码修改的决策流程"""

    # Step 1: 判断变更类型
    变更类型 = 分析变更类型(变更内容)
    # 可能的值：数据结构变更、接口变更、算法变更、参数调整、文档修正

    if 变更类型 in [None, "文档修正"]:
        # 不涉及数据结构和接口的变更，可以直接修改
        print(f"✅ 变更类型：{变更类型 or '简单修改'}")
        执行代码修改(文件路径, 变更内容)
        return "修改完成"

    # Step 2: 影响分析
    print(f"📋 变更类型：{变更类型}")
    受影响的文档 = []

    if 变更类型 == "数据结构变更":
        提取的数据结构 = 提取数据结构名称(变更内容)
        受影响的文档 = find_all_references(提取的数据结构, "doc/")
        print(f"影响分析：数据结构 '{提取的数据结构}' 被以下文档引用：")
        for doc in 受影响的文档:
            print(f"  - {doc}")

    elif 变更类型 == "接口变更":
        提取的接口 = 提取接口名称(变更内容)
        受影响的文档 = find_all_references(提取的接口, "doc/")
        print(f"影响分析：接口 '{提取的接口}' 被以下文档引用：")
        for doc in 受影响的文档:
            print(f"  - {doc}")

    elif 变更类型 in ["算法变更", "参数调整"]:
        # 读取文件对应的模块文档
        对应文档 = 查找对应文档(文件路径)
        if 对应文档:
            受影响的文档 = [对应文档]
            print(f"影响分析：需要更新文档 {对应文档}")

    # Step 3: 询问用户是否继续
    if 受影响的文档:
        print("\n建议执行步骤：")
        步骤编号 = 1
        if 变更类型 == "数据结构变更":
            print(f"  {步骤编号}. 更新 doc/02-技术规范/数据结构规范.md")
            步骤编号 += 1
        elif 变更类型 == "接口变更":
            print(f"  {步骤编号}. 更新 doc/02-技术规范/接口规范.md")
            步骤编号 += 1

        print(f"  {步骤编号}. 更新受影响的模块文档版本历史")
        步骤编号 += 1
        print(f"  {步骤编号}. 修改代码实现")
        步骤编号 += 1
        print(f"  {步骤编号}. 更新测试用例")

        用户确认 = 询问用户("是否按此顺序执行？")
        if not 用户确认:
            return "等待用户确认"

    # Step 4: 执行更新
    当前步骤 = 1

    if 变更类型 == "数据结构变更":
        print(f"步骤 {当前步骤}/{步骤编号}：更新数据结构规范")
        更新数据结构规范(变更内容)
        当前步骤 += 1

    elif 变更类型 == "接口变更":
        print(f"步骤 {当前步骤}/{步骤编号}：更新接口规范")
        更新接口规范(变更内容)
        当前步骤 += 1

    if 受影响的文档:
        print(f"步骤 {当前步骤}/{步骤编号}：更新模块文档版本历史")
        for doc in 受影响的文档:
            追加版本历史(doc, 变更类型, 变更内容)
        当前步骤 += 1

    print(f"步骤 {当前步骤}/{步骤编号}：修改代码")
    执行代码修改(文件路径, 变更内容)
    当前步骤 += 1

    print(f"步骤 {当前步骤}/{步骤编号}：更新测试")
    更新测试用例(文件路径, 变更内容)

    print("\n✅ 所有变更已完成")
    print("建议：运行测试验证变更无副作用")
    return "修改完成"
```

### 8.3 文档一致性检查流程
```python
def 一致性检查(触发原因="手动触发"):
    """AI 执行文档一致性检查"""

    print(f"🔍 执行文档一致性检查（原因：{触发原因}）")
    检查结果 = []

    # 检查 1: 数据结构引用一致性
    print("\n检查项 1: 数据结构引用一致性")
    for 模块文档 in glob("doc/03-模块设计/*.md"):
        引用的数据结构 = 提取数据结构引用(模块文档)
        for ds in 引用的数据结构:
            if not exists_in("doc/02-技术规范/数据结构规范.md", ds):
                检查结果.append(f"❌ {模块文档} 引用的 {ds} 在数据结构规范中不存在")

    # 检查 2: 接口引用一致性
    print("检查项 2: 接口引用一致性")
    for 模块文档 in glob("doc/03-模块设计/*.md"):
        引用的接口 = 提取接口引用(模块文档)
        for api in 引用的接口:
            if not exists_in("doc/02-技术规范/接口规范.md", api):
                检查结果.append(f"❌ {模块文档} 引用的 {api} 在接口规范中不存在")

    # 检查 3: 临时文件位置
    print("检查项 3: 临时文件位置")
    for 文件 in glob("temp/**/*"):
        if 是正式文件(文件):
            检查结果.append(f"⚠️  临时文件 {文件} 应该移到正式目录")

    # 检查 4: 版本历史完整性
    print("检查项 4: 版本历史完整性")
    for 文档 in glob("doc/**/*.md"):
        if not contains(文档, "## 版本历史"):
            检查结果.append(f"⚠️  {文档} 缺少版本历史章节")

    # 报告结果
    print("\n" + "="*50)
    if 检查结果:
        print("一致性检查结果：")
        for 结果 in 检查结果:
            print(f"  {结果}")
        print(f"\n发现 {len(检查结果)} 个问题")
        return "检查未通过"
    else:
        print("✅ 一致性检查通过")
        return "检查通过"
```
