## 反模式 1：文档重复定义数据结构

### ❌ 错误做法

**doc/03-模块设计/用户认证模块.md**：
```python
class UserAuthInput:
    username: str
    password: str
```

**doc/03-模块设计/用户管理模块.md**：
```python
class UserAuthInput:  # 重复定义！
    username: str
    password: str
```

**问题**：
- 数据结构在多处重复定义
- 修改时容易遗漏，导致不一致
- 违反"单一事实来源"原则

### ✅ 正确做法

**doc/02-技术规范/数据结构规范.md**（唯一定义）：
```python
class UserAuthInput:
    """用户认证输入数据结构"""
    username: str
    password: str
```

**doc/03-模块设计/用户认证模块.md**：
```markdown
## 2. 输入输出数据规范

### 2.1 输入
引用：[UserAuthInput](../02-技术规范/数据结构规范.md#userauthinput)

**约束条件**：
- username 不能为空
- password 长度 ≥ 8
```

**doc/03-模块设计/用户管理模块.md**：
```markdown
## 2. 输入输出数据规范

### 2.1 输入
引用：[UserAuthInput](../02-技术规范/数据结构规范.md#userauthinput)
```

```
doc/
```

## 反模式 3：版本历史缺失影响范围

### ❌ 错误做法

```markdown
## 版本历史

| 版本 | 日期 | 说明 |
|------|------|------|
| v1.1 | 2026-03-03 | 修改了用户名字段 |
```

**问题**：
- 未说明修改了什么（类型？具体变更？）
- 未列出受影响的模块
- 其他开发者无法快速评估影响

### ✅ 正确做法

```markdown
## 版本历史

| 版本 | 日期 | 说明 |
|------|------|------|
| v1.1 | 2026-03-03 | [数据结构变更] 修改 UserAuthInput.username 字段类型：str → EmailStr。<br>**影响范围**：用户认证模块、用户管理模块、权限验证模块 |
```

**变更类型标签**：
- `[数据结构变更]` - 数据结构字段增删改
- `[接口变更]` - 接口签名变更
- `[算法变更]` - 核心算法修改
- `[参数变更]` - 参数调整
- `[文档修正]` - 文档错误修正
- `[架构变更]` - 架构调整

## 反模式 4：跳过影响分析直接修改代码

### ❌ 错误做法

```
用户：把密码字段从 str 改为 HashString

AI：好的，我来修改代码。
[直接修改 src/auth/models.py]
[直接修改 src/auth/validator.py]
完成！
```

**问题**：
- 未检查数据结构规范是否需要更新
- 未检查哪些模块受影响
- 可能导致其他模块崩溃

### ✅ 正确做法

```
用户：把密码字段从 str 改为 HashString

AI：检测到数据结构变更请求。

影响分析：
- 变更类型：数据结构变更
- 影响的数据结构：UserAuthInput.password
- 受影响的文档：
  1. doc/02-技术规范/数据结构规范.md（需要更新）
  2. doc/03-模块设计/用户认证模块.md（需要更新版本历史）
  3. doc/03-模块设计/用户管理模块.md（可能受影响）
  4. doc/03-模块设计/权限验证模块.md（可能受影响）

建议执行步骤：
1. 更新数据结构规范
2. 检查并更新受影响的模块文档
3. 修改代码实现
4. 更新测试用例

是否按此顺序执行？
```

## 反模式 5：文档与代码不一致

### ❌ 错误做法

**doc/02-技术规范/接口规范.md**：
```python
def authenticate(username: str, password: str) -> bool:
    """用户认证"""
    pass
```

**src/auth.py**（实际代码）：
```python
def authenticate(username: str, password: str, token: str) -> bool:
    # 实际代码多了一个 token 参数！
    pass
```

**问题**：
- 文档与代码不一致
- 其他开发者按照文档调用会出错
- 违反"文档即规范"原则

### ✅ 正确做法

**情况 1：代码正确，文档过时**
```
AI：⚠️ 检测到文档与代码不一致
- 文档：authenticate(username, password)
- 代码：authenticate(username, password, token)

文档是否需要更新？
[用户确认后]
✅ 已更新接口规范，添加 token 参数
✅ 已更新版本历史
```

**情况 2：文档正确，代码过时**
```
AI：⚠️ 检测到文档与代码不一致
- 文档：authenticate(username, password)
- 代码：authenticate(username, password, token)

根据规范，以文档为准。是否修改代码以匹配文档？
[用户确认后]
✅ 已修改代码，移除 token 参数
```

## AI 自检清单

在执行任何操作前，AI 应该自问：

- [ ] 是否创建了新文档？→ 是否使用了模板？
- [ ] 是否修改了数据结构？→ 是否更新了数据结构规范？
- [ ] 是否修改了接口？→ 是否更新了接口规范？
- [ ] 是否修改了代码？→ 是否先更新了文档？
- [ ] 是否创建了临时文件？→ 是否放在 temp/ 目录？
- [ ] 是否修改了文档？→ 是否更新了版本历史？
- [ ] 是否涉及多个模块？→ 是否进行了影响分析？
