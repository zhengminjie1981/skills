# 质量门禁规范

## 编码前门禁（Gate-1）

**必须通过**：
- [ ] 需求已明确，功能边界清晰
- [ ] 系统架构设计已完成并评审通过（如涉及架构变更）
- [ ] 数据结构规范已定义并更新（如涉及数据结构变更）
- [ ] 接口规范已定义并更新（如涉及接口变更）
- [ ] 模块设计文档已完成（包含测试用例）
- [ ] 文档一致性检查通过（评分 ≥ 95%）
- [ ] 影响分析已完成（列出所有受影响的文档和代码）

**检查方法**：
```bash
# 检查文档完整性
ls -la doc/系统设计/ doc/技术规范/ doc/模块设计/

# 检查版本历史是否存在
grep -l "## 版本历史" doc/**/*.md
```

## 代码提交门禁（Gate-2）

**必须通过**：
- [ ] 代码实现与文档定义一致
- [ ] 单元测试通过（覆盖率 ≥ 80%）
- [ ] 代码审查通过（至少1人审查）
- [ ] 集成测试通过（如涉及接口变更）
- [ ] 文档已同步更新（如有变更）
- [ ] 临时文件已清理或迁移

**检查方法**：
```bash
# 运行测试
pytest tests/ --cov=src/ --cov-report=term-missing

# 检查代码与文档的一致性（手动或脚本辅助）
# 例如：检查函数签名与接口规范是否一致
```

## 发布门禁（Gate-3）

**必须通过**：
- [ ] 所有测试用例通过（单元、集成、系统测试）
- [ ] 测试报告完整且符合验收标准
- [ ] 性能指标达标（如有性能要求）
- [ ] 安全测试通过（如涉及安全相关功能）
- [ ] 文档已更新（README、版本历史、API文档）
- [ ] 项目计划与进展记录已更新
- [ ] 变更日志已更新

**检查方法**：
```bash
# 完整测试
pytest tests/ -v
pytest tests/integration/ -v

# 性能测试
python scripts/performance_test.py

# 文档生成
mkdocs build
```

## 一致性评分标准

### 评分维度

| 维度 | 权重 | 检查内容 |
|------|------|---------|
| 架构一致性 | 25% | 模块边界、数据流与系统架构设计一致 |
| 数据结构一致性 | 25% | 所有模块引用的数据结构与规范一致 |
| 接口签名一致性 | 25% | 接口定义与规范一致 |
| 状态一致性 | 15% | README 状态与实际代码状态一致 |
| 索引完整性 | 10% | 文档索引包含所有现有文档 |

### 评分方法

**手工评分**（推荐用于小型项目）：
1. 按维度逐项检查
2. 发现不一致项扣分（每项扣 5-10 分）
3. 计算总分

**自动化评分**（推荐用于大型项目）：
```bash
# 示例：自动化检查脚本
python scripts/check_consistency.py --project . --output report.json

# 输出：一致性评分和详细报告
```

### 通过标准

- **≥ 95%**：优秀，可以进入下一阶段
- **90-95%**：合格，需修复次要问题
- **< 90%**：不合格，必须修复后重新检查

## 常见质量问题

### 问题1：文档与代码不一致

**症状**：接口规范定义的参数与代码实现不符

**解决**：
1. 立即停止编码
2. 更新文档或代码（以文档为准）
3. 重新执行一致性检查

### 问题2：缺少版本历史

**症状**：文档没有版本历史章节

**解决**：
```markdown
## 版本历史

| 版本 | 日期 | 说明 |
|------|------|------|
| v1.0.0 | 2026-03-02 | 初始版本 |
```

### 问题3：临时文件污染正式目录

**症状**：`doc/` 或 `src/` 中出现调试脚本

**解决**：
```bash
# 移动到临时目录
mv doc/debug_script.py temp/scripts/

# 或删除
rm doc/debug_script.py
```

## 质量门禁流程图

```
开始
  ↓
[Gate-1] 编码前检查
  ├─ 通过 → 开始编码
  └─ 不通过 → 返回完善文档
  ↓
[Gate-2] 提交前检查
  ├─ 通过 → 提交代码
  └─ 不通过 → 修复问题
  ↓
[Gate-3] 发布前检查
  ├─ 通过 → 发布
  └─ 不通过 → 返回修复
  ↓
完成
```
